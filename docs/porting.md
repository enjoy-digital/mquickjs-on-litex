# Porting Notes

This page is for changing the firmware. For normal use, start with
[simulation.md](simulation.md) or [hardware.md](hardware.md).

## Runtime Shape

`firmware/main.c` does very little:

1. Initialize the LiteX UART.
2. Print the banner and CPU/heap information.
3. Create a mquickjs context over a fixed heap buffer.
4. Evaluate the embedded JavaScript script.
5. Print `[mqjs] done` or `[mqjs] fail`.

mquickjs does not call `malloc`. The firmware gives it one aligned heap:

```c
static uint8_t mqjs_heap[LITEX_MQJS_HEAP_SIZE] __attribute__((aligned(16)));
```

The default heap is 1 MiB. Override it when building:

```sh
make sim HEAP_SIZE=2097152
make firmware HEAP_SIZE=2097152
```

Add `MEMORY_DUMP=1` when you want the mquickjs memory summary.

## Embedded Script

`tools/embed_script.py` converts a `.js` file into `build/user_script.h`.
The generated buffer has one extra `0x00` byte after the script. Keep
that sentinel: the mquickjs parser does one byte of lookahead past the
explicit source length.

## LiteX Bindings

The JavaScript `litex` object is generated by
`firmware/mqjs_stdlib_litex.c`. The low-level CSR access lives in
`firmware/mqjs_port.c`.

To add a binding:

1. Add the C helper in `firmware/mqjs_port.c`.
2. Declare it in `firmware/mqjs_port.h`.
3. Register the JavaScript function in `firmware/mqjs_stdlib_litex.c`.
4. Exercise it from `examples/demo.js` or `examples/sdcard/main.js`.

Bindings should degrade gracefully when a CSR is absent. For example,
LED writes are no-ops if there is no LED CSR, and switch/button reads
return zero if those CSRs are missing.

## Generated Stdlib Headers

The firmware build compiles `firmware/mqjs_stdlib_litex.c` once as a
host tool, then runs it to generate:

| Header | Used by |
|--------|---------|
| `build/mquickjs_atom.h` | `third_party/mquickjs/mquickjs.c` |
| `build/mqjs_stdlib.h` | `firmware/main.c` |

The generator is run with `-m32` so the tables match the rv32 firmware
layout even on a 64-bit host.

The Makefile removes a stale upstream `third_party/mquickjs/mquickjs_atom.h`
before compiling mquickjs. This keeps the target build using the LiteX
stdlib atoms, not a host-side upstream table.

## SDCard Support

`litex.readFile(path)` and `litex.load(path)` are available when the SoC
has `CSR_SDCARD_BASE` or `CSR_SPISDCARD_BASE`.

Without SDCard hardware, these functions throw a JavaScript exception.
That keeps the same firmware sources usable in simulation, on minimal
boards, and on SDCard-capable boards.

## Linker Layout

`firmware/linker.ld` places the firmware in LiteX `main_ram`, not in the
small integrated SRAM. This leaves room for the script, stack and
mquickjs heap.

Keep section alignment attached to the output section definition. Moving
alignment padding inside `.data` can break the crt0 copy from ROM to RAM
and make early `printf` output disappear.

## Not Ported

The upstream `setTimeout`/`clearTimeout` and terminal-colour REPL pieces
are intentionally left out. The demo stays small and synchronous; the
hardware-facing parts are the interesting bit here.
